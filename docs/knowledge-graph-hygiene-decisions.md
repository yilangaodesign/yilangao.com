# Knowledge Graph Hygiene Decisions

> **Purpose:** Record the decisions made during Plan D (Graph Hygiene Remediation) so future agents and contributors understand *why* the graph is shaped the way it is, not just *what* it looks like.
>
> **Status:** LOCKED. D3 and D4 may proceed.
>
> **Context:** Five structural gaps (G1-G5) were surfaced during plan review. This document records the decision for each. The empirical evidence comes from the D1 hygiene-audit run (`scripts/audit-graph-hygiene.mjs`).

---

## G1: Edge Type Hierarchy

**Question:** Should the six edge types (`enforces`, `supersedes`, `documents`, `derivedFrom`, `triggers`, `references`) have a documented authority ranking, or should they be explicitly peer-level?

**Decision: Peer-level (no ranking).**

**Rationale:**
- No script, tool, or retrieval path currently consumes an authority rank.
- Adding a rank column to §5 without a consumer creates spec weight for zero functional benefit.
- The semantic distinction (e.g., `enforces` is "stronger" than `references`) is already implicit in the type name; agents can reason about it contextually.
- If a future plan adds weighted retrieval (e.g., prioritizing `enforces` edges over `references` in the MCP server's subgraph results), a ranking table can be introduced at that time with empirical calibration data.

**Action for D3:** Add a one-line disclosure to §5 stating that edge types are peer-level with no authority ordering.

## G2a: Implicit Synthetic Types (route, guardrail, section)

**Question:** The build script auto-generates nodes of type `route`, `guardrail`, and `section` from AGENTS.md anchors and per-document section headings. These types are not declared in §8. Should they be promoted to official types or documented as synthetic?

**Empirical evidence (D1):**
- `route`: 24 nodes (auto-derived from AGENTS.md route anchors)
- `guardrail`: 43 nodes (auto-derived from AGENTS.md guardrail anchors)
- `section`: 133 nodes (auto-derived from per-document section headings with anchors)

**Decision: Document as synthetic in a new §8.1.**

**Rationale:**
- These types have no source frontmatter — they are build artifacts derived from anchor patterns. Promoting them to the official §8 table (which implies frontmatter authoring) would be misleading.
- A new §8.1 "Build-derived synthetic node types" clearly separates authored types (§8) from inferred types (§8.1), making the taxonomy honest about its layering.
- The `checkFrontmatterSchema` audit in `audit-docs.mjs` can stop silently bypassing these types once §8.1 documents them — they become recognized, not bypassed.

**Action for D3:** Add §8.1 to `docs/knowledge-graph.md` listing `route`, `guardrail`, and `section` as synthetic types with a note that they are generated by `scripts/build-graph.mjs` from anchor patterns, not from frontmatter.

## G2b: The `unknown` Type

**Question:** Two nodes in the graph have type `unknown` (the two renumber logs with `node_type:` instead of `type:`). Should `unknown` be promoted to an official type?

**Empirical evidence (D1):**
- `unknown`: 2 nodes (`docs/design-renumber-log.md`, `docs/eng-renumber-log.md`) — both are caused by the field-name drift (`node_type:` instead of `type:`), so the build script can't parse their type.

**Decision: Keep `unknown` as a build-warning fallback. Do NOT promote.**

**Rationale:**
- `unknown` is an error state, not a legitimate type. It means the build script couldn't determine the type.
- Once D5 fixes the `node_type:` → `type:` field drift, these two nodes will resolve to `renumber-log` and the `unknown` count will drop to zero.
- Document `unknown` in §8.1 as a fallback type (not an authored type) so the schema check knows to bypass it without silently hiding a real problem.

**Action for D3:** Add `unknown` to §8.1 with a note that it is a fallback type indicating a parse failure, not a legitimate classification. The schema check should continue to bypass it (not promote to ERROR).

## G3: Dead-End Detection

**Question:** Should outbound dead-end detection (nodes with zero outgoing edges) be advisory-only in the hygiene script, added to `audit-docs.mjs` as WARN, or gated as ERROR?

**Empirical evidence (D1):**
- 151 nodes with zero outbound edges: 56 `feedback`, 95 `section`.
- Both are expected terminal types — feedback entries cite inward but rarely outward, and section anchors are leaf navigation targets.

**Decision: Option B — add to `audit-docs.mjs` as WARN with per-type allowlist.**

**Rationale:**
- Feedback and section nodes are legitimately terminal. A per-type allowlist (`feedback`, `section`, `release`, `eval-baseline`) prevents noisy false positives.
- Running as WARN in the default audit surfaces any *unexpected* dead-end types (e.g., a spoke or hub with zero outbound edges would be a real problem) without blocking CI.
- D1's hygiene audit script provides the advisory one-off check; the audit-docs integration makes it part of the standard workflow.

**Action for D4:** Add `checkOutboundDeadEnds()` to `scripts/audit-docs.mjs` at WARN level. Allowlist: `feedback`, `section`, `release`, `eval-baseline` are expected to have zero outbound edges. Types like `spoke`, `hub`, `skill`, `rule`, `anti-pattern` should flag as WARN if they have zero outbound edges.

## G4: Node Lifecycle

**Question:** Should `deprecated: true` be extended with temporal metadata (`addedAt`, `deprecatedAt` ISO dates) and a cascade rule (warn on edges to deprecated nodes)?

**Empirical evidence (D1):**
- 1 deprecated node (`release-log-archive`), 24 edges pointing to it.
- The edges are mostly `documentedBy` inverses from release entries — structurally expected since archived entries were documented by the archive file.

**Decision: Keep single-bit `deprecated` for now. Add cascade-warning rule.**

**Rationale:**
- Only 1 deprecated node exists today. Adding ISO date fields to the schema and build pipeline for 1 node is premature complexity.
- The cascade-warning rule (flagging edges TO deprecated nodes) is useful even without dates — it catches stale citations. D1 already surfaces these in subcheck 5.
- If the number of deprecated nodes grows, a future plan can add temporal fields. The schema allows adding `deprecatedAt` as an optional field later without breaking anything.

**Action for D3:** Add a cascade-deprecation note to §16 maintenance rules: "Edges pointing to deprecated nodes should be reviewed. The audit script flags these as WARN."

**Action for D4:** The `checkOutboundDeadEnds` or a separate check can flag edges to deprecated nodes at WARN level. (D1's subcheck 5 already does this — consider folding into audit-docs.)

## G5: ID-Format Drift

**Question:** The spec's §3 ID-naming table says hubs should be `hub-{pillar}` and spokes should be `spoke-{slug}`. Reality: hubs use bare names (`design`, `engineering`, `content`) and spokes use `{pillar}-{slug}` (e.g., `design-typography`). Additionally, two renumber logs use `node_type:` instead of `type:`, and `docs/eval-results.md` embeds a run ID in its node ID.

**Empirical evidence (D1):**
- 3 hubs fail the `hub-*` pattern (they use bare names: `design`, `engineering`, `content`)
- 57 spokes fail the `spoke-*` pattern (they use `{pillar}-{slug}`)
- 4 routes fail `route-\d+` pattern (they use `route-knowledge-{a,b,c,d}`)
- Zero files in the entire repo use `hub-` or `spoke-` prefixed IDs.
- 2 files use `node_type:` instead of `type:`

**Decision: Option B — update spec to match reality.**

**Rationale:**
- The entire codebase consistently uses bare hub names and `{pillar}-{slug}` spoke names. Every citation, every cross-reference, every build artifact uses these patterns. Migrating to the spec's `hub-`/`spoke-` convention would require renaming ~60 IDs plus grep-replacing hundreds of citations for zero functional benefit.
- Option A (migrate to spec) was empirically confirmed as inverted — it would break everything that works.
- Option B is ~5 line changes to §3 and produces zero churn.

**Actions for D3:**
1. §3 ID-naming table: Change hub pattern from `hub-{design,engineering,content}` to `{design,engineering,content}` (bare pillar name). Change spoke pattern from `spoke-{slug}` to `{pillar}-{slug}`.
2. §3 ID-naming table: Add row for route sub-type `route-knowledge-{a,b,c,d}` to acknowledge the knowledge-route naming convention.
3. §3 ID-naming table: Add row for `eval-results` as a standalone ID for eval report nodes (drop the run qualifier from the ID).

**Actions for D5:**
1. Rename `node_type:` to `type:` in `docs/eng-renumber-log.md` and `docs/design-renumber-log.md`.
2. Change `id: eval-results-full-001` to `id: eval-results` in `docs/eval-results.md` and add `eval_run_id: full-001`.
