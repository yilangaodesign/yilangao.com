# Project Dossier: ETRO Framework

## Current State

- **Thesis:** Trust architecture has two axes, not one: the four questions every user asks of a machine-made decision (Explainability, Traceability, Reversibility, Observability — present, past, future, always), and the volume at which the system answers them. ## Style Preferences

- **Article format** uses the case study page anatomy (hero, sidebar, intro blurb, content blocks) but with longer body text per section.
- **Blurb body contract (essay, not case study).** The blurb body IS the lede, not a trailer. The CEO-dismantle scene (5 paragraphs: direct quote + 4 narrative paragraphs) lives in `introBlurbBody`, not in a Section 1 content block. Article opens at the backstory section ("The Success That Almost Wasn't."). Do NOT re-add a standard case-study-style 2-paragraph trailer - that produces two openings. The scope statement (`description`) must not recap any beat that now lives in the blurb; state the underlying fact instead (e.g., "The framework has a second axis" rather than "A Blackstone veteran challenged the approach...").
- **Section titles** use the same named techniques as case studies: Object Substitution, Framework Inversion, Verdict + Staccato.
- **Blurb headline** (hook) is the essay's actual title: "Your AI product doesn't need more transparency. It needs the right kind." Uses Verdict-as-Headline (Technique 8) — the essay's central claim stated as the title. The phrase "Trust has a shelf life. Sometimes." (Deliberate Ambiguity) is retained as Section 7's heading (the closer), not the hook. Rule: the essay title goes in `introBlurbHeadline`; evocative closer phrases stay inside the body.
- **Voice register:** Warmer than case study Zone 3 (closer to sustained Zone 1 energy). Columnist voice rather than restrained declarative.
- **Inline bold in body: never.** Per `voice-style.md` §13.1 and CAP-030: bold is reserved for H2 section headings and impact metrics. When materializing from a user-supplied markdown source that uses inline `**bold**` as scan-anchor typography, strip all of it. Convert scan-anchor labels to colon-prefixed paragraph starts ("Per user:", "Scaffolding:"); convert sentence-level emphasis to plain prose; convert tonal emphasis to italic (`*word*`, not banned). Also do not use inline `###` headings inside `bodyMarkdown` - `lexicalToHtml` drops heading nodes when mixed with paragraphs in the same richText root. Use separate `heading` blocks or restructure the paragraph.
- **Redaction strategy:** All startup names removed. Goldman Sachs handled with wry acknowledgment ("a financial institution I probably can't name"). Internal product name (Meteor) kept in the Meteor case study but not repeated here.
